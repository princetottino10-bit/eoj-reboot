## 2. 判定の作法(最重要)

### 帯で判定する。点で判定しない
- 合否は必ず**KPI帯**(kpi-design記載)との比較で言う。「良くなった/悪くなった」という相対評価だけで判断しない
- 帯の外でも「どの向きに・どれだけ外れたか」を書く。境界ぎりぎりは「帯内だが監視点」と明記する(例: 手詰まり19.3%は帯上限20%ぎりぎり→監視点として記録済み)

### 誤差を常に添える
- **2000試合の先手勝率は±2.2pt(95%CI)揺れる。実測済み**(16バッチのレンジ47.95〜51.90%。`r1-firstplayer-32k-report.md`)
- ±2pt以内の差を根拠に優劣を語らない。先手勝率で合否判定するランは8000試合以上(±1.1pt)を要件とする
- 層別集計はnを必ず併記。n<50の層は参考値扱い

### 極端な数字が出たら、結論より先に実験設計を疑う
- 過去の実例: fork計測v1でΔ−20.7ptという極端な値が出た→設計バグだった(介入の非対称)。「ミラー戦で介入なしなら勝率50%になるはず」という**対称性の自己検証**をゲートに入れて v2 で解決した(`react-value-report.md` §1に顛末)
- カウンターファクチュアル比較では、両世界の介入量を対称にする。自己検証(null条件で50%)をレポート冒頭に置き、FAILなら結果を読まずに停止する

### 相関と因果を区別する
- 観測ログの「使った/使わない比較」には選択バイアスが乗る(劣勢な人ほど使う→「使うと負ける」に見える。傘の誤謬)
- 因果を言いたいときはfork法(同一局面から分岐・同一乱数系列でペア比較)+層別(優劣×王手状況の6層)

### ノブは1回に1つ
- 1ランで動かす変数は原則1つ。2つ動くと寄与が分離できない
- 分離が必要になったら切り分けラン(D1方式: 片方だけ戻して差分を測る)を挟む。実例: 形状導入で壊れた際、「形の広さ」と「反撃ルール」をD1で分離→主因は形の側と判明(反撃ルールは増幅装置)

### ノブの効き目は環境依存。加算近似は乱戦経済で壊れる(v4切り分けの教訓)
- 過去の掃引で得たノブの効果量(例: dm1段≈5〜7pt)は、**同じ環境でのみ有効**。ゲーム構造が変わった環境(撃破経済が循環する乱戦など)では通用しない。v4切り分けで実証: dm2→1の先手勝率寄与は予測5〜7ptに対し実測−0.9pt
- 多ノブ環境では単独復元の効果を足しても全体変化を説明できない(相互作用支配)。実例: 4ノブ復元の合計が先手−1.6pt(全体差−6.4pt)、生命勝ち−2.4pt(同−20.9pt)。**単独寄与は組み合わせ選定の初期近似にのみ使い、候補構成は必ず1本として再実測する**

### 実装とルール文言の乖離は最優先で検出・測定する(B1事件)
- シミュ実装が紙ルールと食い違っていると、**全ベースラインが「別のゲーム」の測定になる**。実例: 召喚隣接判定が実装=自軍のみ/紙=敵味方問わずで乖離しており、紙ルールでは平均+4.1R・撃破+5.4・弱点撃破率+13.7ptと構造レベルで別物だった(先手勝率だけは不変)。人間プレイテストとの答え合わせは紙ルール準拠の測定でしか成立しない
- 新しい計測モードを作るとき、実装ノートに「ルール文言との差分」を必ず自己申告する(codexがv4プローブでこれをやって発覚した。良い前例)

## 3. レポートの作法

- **結論を先に**(§0に採否・数値・一言)。いきさつ→数値→含意の順
- **個人名は書かない**(チーム内の発案は「チーム議論から」と書く)
- **失敗も書く**。破棄した計測は破棄バナー付きで残し、なぜ間違えたかを教訓として本文に含める(信頼性の担保になる)
- 提案と確定を明確に分ける(Status行: 提案(未確定)/確定)。採用判定はユーザー+チームが行う。レポートは判定材料の提示まで
- 誤解されやすい指標には説明を添える。実例:「返し手勝率46.9%」は「返すと負けやすい」ではなく患者層の拡大(返しが安くなる→弱い局面でも返せる→平均が薄まる)。この手の誤読はチームで実際に起きた
- 数値の変更を伴う提案には「暫定か恒久か」の線引きを明記(例: 7/18のHP+1は暫定調整であり、基盤v3.1の変更ではない)
- 保存形式: ローカルmd(docs/)+必要ならGoogleドキュメント化してチーム共有

## 4. 実行・記録の規約

- seed `20260702` 固定、シード系列 `seed + gameIndex * 9176`、check-search depth 6

## 7. KPI帯の改訂について(引き継ぐ判断基準)

KPI帯を動かすのは「実測が帯に合わない」ときではなく、**「ゲームのデザイン意図が変わった」とき**。実測が帯を外れたら、まず構成側を直すのが原則(帯は目標であって記述ではない)。

改訂が正当なのは:
- ルール・レイヤーの追加で指標の意味が変わったとき(例: 形状導入で「弱点撃破率」の分母構造が変わる)
- 体感テストでデザイン意図側が更新されたとき(例: 7/18で「9Rは長い」となれば長さの帯自体を見直す)

改訂時は `kpi-design-pure-vanilla.md` の改訂履歴に「いつ・なぜ・どの帯をどう変えたか」を必ず追記する。帯の変更履歴が消えると、過去レポートの合否が読めなくなる。

## 8. 進め方のリズム(参考)

1. 問い(1つ)を決める → 判定基準を**先に**固定(どうなったら合格/不合格か)
2. タスク指示書を書く(構成固定・計測項目・保存先・「判定はしない、保存まで」)
3. 実行 → 結果を帯と誤差込みで精読 → 異常があれば設計を疑う
4. レポート化(結論先出し・個人名なし・失敗込み) → チーム判断を仰ぐ
5. 採用されたら: 凍結スナップショット化、関連ドキュメント(README索引・overview・roadmapトラッカー)を同期、破棄物にバナー
