## フェーズ1: キャリブレーション(仕様準拠版)

`docs/kpi-design-pure-vanilla.md` の段階1を仕様準拠エンジンで再実行する。

- 純粋素体(前方1マス/1対象/後方弱点/物理のみ)。属性は新5属性を均等割当、盤面は推奨配置
- 4カーブ(再行動 heavy/standard/light の3点比較を含む)× コスト分布5候補、ミラー対戦
- `DESTROY_MANA_GAIN` は暫定値(例: 1)で固定し、フェーズ2で感度を見る
- スコア関数なし。全KPIの値域観測が目的

完了条件: 値域レポートを基に、行動量KPI・カーブ健全性KPIの帯を合意して確定。スコア関数を新KPI体系で書き換え。

主要リスク: 墓地リシャッフル(デッキ切れなし)により長期戦の終了圧が消えるため、P90と引き分け率が旧シミュより悪化する可能性が高い。悪化幅を最初に観測し、対策(生命価・DESTROY_MANA_GAIN・秘術)の必要量を見積もる。

## フェーズ2: 本探索 — constants とカーブの決定

探索対象(=このフェーズの成果物):

| 種別 | 対象 |
|---|---|
| constants | `START_LIFE` / `MANA_PER_TURN` / `START_MANA_FIRST` / `START_MANA_SECOND` / `MAX_HAND` / `DECK_SIZE` / `WIN_CONTROL` / `BLIND_SPOT_BONUS` / `ATTRIBUTE_HP_BONUS` / `DESTROY_MANA_GAIN` / `MAX_HP` /(採用なら `MAX_MANA`) |
| カーブ | ステータスカーブ(HP/ATK/再命令コスト)、生命価カーブ、スタンダードコスト分布(中心+許容半径) |

進め方:

1. 既往の知見を初期値として引き継ぐ(ライフ15、収入+3、後手+1、弱点+2、属性±2、5マス制圧、手札5)。ただし全て「形状込み旧環境の参考値」であり、仕様準拠の純粋環境で再検証する
2. カーブ×コスト分布のグリッドを新スコア関数で選定
3. **素体アイテム**(除去/経済/強化/ドローの4原型×固定8枠)をデッキに含めて分布を決める(デッキ力学を実ゲームに近づけるため。素体アイテムの性能は地味に固定し、値付けはフェーズ4)
4. 新レバーの感度測定: `DESTROY_MANA_GAIN`(0/1/2)、`MAX_HP`(上限なし/カーブ最大+2/固定10等)
5. 頑健性テスト: スタンダード分布±2枚シフトの近傍でKPI維持、ズレ対戦の勝率≤55〜60%

完了条件: constants提案v1と各カーブを確定し、スナップショット凍結(従来の作法: シード・SHA・再現コマンド)。RULE_SPEC.md の constants 章に値を記入できる状態。

## フェーズ3: 形状の値付け

純粋素体を基準に、形状要素を1変数ずつ足して勝率・KPI影響を測る差分実験。

- 攻撃範囲の拡張(前2マス、前左右、全周…)、対象数(1→複数同時)、術式(全域・反撃不可・死角無効)、反撃範囲・死角パターンの変形
- 結果から**形状税(またはコスト換算表)を経験的に再導出**する(現行の形状税は経験則なので置き換える)

完了条件: 「形状→コスト/ステータス補正」の換算表。実カードの形状に価格を付けられる状態。

## フェーズ4: 効果・霊具レイヤー

- 素体アイテムを実霊具に置き換えながら、効果の原型(除去/加速/移動/向き操作/強化/ドロー)ごとに値付け
- 式神の効果・キーワード(連撃・挑発など)を素体との差分として値付け
- スペル/効果を扱うAIの拡張をセットで実施(AIが使えない要素は「弱い」と誤計測される)
- 監視重点: チェック返し手段の構成比の変化、旧環境で見えた膠着セル(耐久系ミラー)の解消度、P90

完了条件: 効果込みでKPI帯を維持。効果の値付けガイドライン文書化。

## フェーズ5: 秘術の設計と統合

RULE_SPEC §13(生命を消費する大技、詳細未定)の設計をバランス面から支援する。

- 生命コストの相場設計: 生命は敗北条件かつ(生命価経由で)撃破ペナルティなので、秘術コストは「制圧レースを何ターン分犠牲にするか」として換算できる
- 発動条件・回数の案をシミュで比較(旧ウルトの「相手3体以上・1ゲーム1回」は参考の出発点)
- 長期戦の終了圧としての役割も評価(フェーズ1で観測する膠着リスクの対策候補)

完了条件: 秘術ルールの改訂案+バランス根拠。RULE_SPEC §13 の確定に反映。

## フェーズ6: 実カードプールへの適用

- `cards.json` を新体系に改訂: 属性の再マッピング(火水土木無→地水火風空)、生命価の付与、カーブ+形状税+効果値付けからの機械生成(`cards.curve-draft.json`)
- カーブ準拠バリデータ(偏差は例外表経由のみ)
- 検品アプリで全カード確認
- プール構成の見直し: コスト5〜6の空洞(現状全70枚中4枚)を、スタンダード分布が要求する枚数と突き合わせて埋める
- 派閥別デッキでの相性行列(十分なサンプル数で)

完了条件: 新cards.jsonドラフトの完成。派閥間勝率の許容帯(例: 40〜60%)を満たす。

## フェーズ7: ドラフトフォーマットと人間プレイテスト

- 陣営デッキドラフトの方式を具体化(提示数・選択方式・BAN等)。スタンダード分布の「許容半径」は、各陣営デッキのカーブ個性を振ってよい設計幅として使う
- 人間プレイテストでシミュKPIを実戦校正: 特に「4チェック返し率はシミュより実戦で下がる」仮説の検証、体感の長さと平均Rの対応
- KPI帯の最終調整とバランスパッチ運用ルール(変更→シミュ回帰→KPI表で判定、の定常プロセス化)

完了条件: フォーマット確定。バランス調整が定常運用(変更はKPI回帰テストを通す)に移行。

## 横断原則(全フェーズ共通)

1. **RULE_SPEC.md が正本。** シミュと仕様が食い違ったら、シミュを直すか仕様を改訂するかを明示的に決める(黙って乖離させない)
2. **1フェーズ1目的。** 交絡を避けるため、基盤(数値)→形状→効果→秘術の順に1レイヤーずつ値付けする
3. **AIは計測器。** 要素追加のたびにAI拡張をセットで行い、オラクル実現率で計測器の健全性を確認する
4. **凍結の作法。** 各フェーズの完了時にシード・SHA・再現コマンド付きでスナップショットを残し、次フェーズは前フェーズ凍結版を対照群にする
5. **KPIは目的関数に必ず反映。** スコア関数と KPI 表の乖離(旧目標のハードコード事故)を再発させない
